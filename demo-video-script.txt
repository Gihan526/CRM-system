CRM Lead Management System — Demo Video Script
===============================================
Total time: ~9–10 minutes
Record at a natural pace. Pause to let the screen load.


INTRO (0:00 – 0:45)
-------------------
"Hi, I'm [Your Name], and this is my full-stack CRM Lead Management System.
This is a web application built for a small sales team to track leads through a
sales pipeline, add notes, and view a dashboard of key metrics.

The stack is React with TypeScript on the frontend, Express.js with TypeScript
on the backend, SQLite with Prisma ORM for the database, and Better Auth for
authentication. Let me show you how it works."


PART 1: RUNNING THE PROJECT (0:45 – 1:15)
------------------------------------------
"First, let me start the project from a clean state. I'll run the database
migrations and seed script to create the test user and demo data."

[Type in terminal:]
cd backend && npx prisma migrate dev && npx prisma db seed

"This creates a SQLite database and seeds 8 demo leads plus a test user.
Now I'll start both the backend and frontend."

[Type in terminal:]
cd .. && npm run dev

"The backend runs on port 3001, and the frontend on port 5173.
Let me open the browser."

[Open browser to http://localhost:5173 — show login page]


PART 2: LOGIN FLOW (1:15 – 1:45)
---------------------------------
"Since I'm not logged in, the app redirects to the login page.
The authentication is handled by Better Auth, which provides email-password
login with secure session cookies. Let me log in with the test credentials."

[Type on login form:]
Email:    admin@example.com
Password: password123
[Click Sign In]

"After login, the session is stored in an HTTP-only cookie, and the frontend
fetches the user session to update the UI. I'm now on the dashboard."


PART 3: DASHBOARD (1:45 – 2:30)
--------------------------------
"The dashboard shows a quick overview of the sales pipeline. We can see
8 total leads, broken down by status: New, Contacted, Qualified, Proposal Sent,
Won, and Lost. We also have two deal value cards showing the total estimated
deal value — $670,000 — and the total won deal value — $200,000.

These numbers are calculated on the backend using Prisma aggregate queries,
so they're always accurate even as leads change."

[Point at each stat card]

"From here I can also quickly add a new lead with this button."


PART 4: CREATING A LEAD (2:30 – 3:15)
--------------------------------------
[Click "Add New Lead"]

"The lead form uses Mantine's form validation. I need to fill in the required
fields: Lead Name, Company Name, Email, Lead Source, and Status."

[Fill in the form:]
Lead Name:     Alex Turner
Company:       Rocket Labs
Email:         alex@rocketlabs.com
Phone:         +1 555-0000
Lead Source:   Website
Status:        New
Assigned To:   Admin User
Deal Value:    85000

[Click "Create Lead"]

"The form submits to the backend via a POST request to /api/leads.
The backend validates the data and creates the record in the SQLite database.
Now I'm back on the leads list."

[Show the new lead in the table]


PART 5: VIEWING & EDITING A LEAD (3:15 – 4:00)
-----------------------------------------------
[Click on a lead — e.g., "Sarah Johnson"]

"Clicking a lead takes me to the detail page. Here I can see all the lead
information in a grid layout, plus I can update the status with these buttons."

[Click a status button — e.g., change "Contacted" to "Qualified"]

"This sends a PUT request to /api/leads/:id with just the status field.
The backend updates the record and returns the updated lead.
The UI refreshes to show the new status badge."


PART 6: ADDING NOTES (4:00 – 4:45)
-----------------------------------
"A key CRM feature is keeping notes on every interaction.
Here I can add a note about a call or meeting."

[Type in the note textarea:]
Had a great call. Alex is interested in the enterprise tier.
Follow up with pricing next week.

[Click "Add"]

"The note is saved to the database with a foreign key linking it to this lead.
It also stores who created it — that's pulled from the authenticated session —
and the timestamp. Notes appear in a timeline view, newest first."

[Show the new note in the timeline]


PART 7: SEARCHING & FILTERING LEADS (4:45 – 5:30)
--------------------------------------------------
[Navigate back to the Leads page]

"The leads page supports filtering by status and source, plus searching by
name, company, or email. The status filter hits the backend API with a query
parameter, while the other filters run client-side on the current dataset."

[Change Status filter to "Contacted"]
"Now we only see contacted leads."

[Clear status filter. Type "acme" in the search box]
"The search instantly filters the table to show only matching leads."

[Clear search]


PART 8: DELETING A LEAD (5:30 – 5:45)
--------------------------------------
"I can also delete leads. Let me delete the lead I just created."

[Find "Alex Turner", click the three-dot menu, click Delete]

"There's a browser confirmation to prevent accidents.
The DELETE request goes to /api/leads/:id, and the backend removes the lead.
Because I set onDelete: Cascade in the Prisma schema, any notes attached to
this lead are automatically deleted too."


PART 9: BACKEND EXPLANATION (5:45 – 7:15)
------------------------------------------
[Switch to VS Code / code editor — backend folder]

"Now let me explain the backend. I used Express.js with TypeScript.
Here's the main server file, index.ts."

[Open backend/src/index.ts]

"The server starts by setting up CORS to allow the frontend to make requests.
Then Better Auth handles all authentication routes at /api/auth/*.
After that, the rest of the API routes are protected by this auth middleware."

[Scroll to authMiddleware in middleware/auth.ts]

"The auth middleware uses Better Auth's getSession method.
It reads the cookie from the request headers, validates it, and attaches the
user object to the request. If there's no valid session, it returns a 401
Unauthorized. This protects all lead, note, and dashboard endpoints."

[Scroll to lead routes]

"The lead routes cover full CRUD: GET for listing with filtering, GET by ID,
POST to create, PUT to update, and DELETE to remove. Each route uses Prisma
to query the SQLite database. For example, the dashboard stats route runs
multiple count and aggregate queries in parallel to get all the numbers at once."

[Open backend/prisma/schema.prisma]

"Here's the database schema. Prisma manages the tables.
Better Auth handles the user, session, account, and verification tables
automatically. I defined two custom tables: Lead and Note.
A lead has many notes, and when a lead is deleted, its notes are
cascade-deleted too."


PART 10: FRONTEND EXPLANATION (7:15 – 8:30)
--------------------------------------------
[Switch to VS Code — frontend folder]

"On the frontend, I used React with Vite and TypeScript.
The UI is built with Mantine, which provides clean, accessible components
out of the box."

[Open frontend/src/App.tsx]

"The app uses React Router for navigation. There's an AuthProvider context
that wraps the entire app. It checks the session on mount using Better Auth's
getSession method and stores the user in React state. The ProtectedRoute and
AuthRoute components handle redirects based on auth status."

[Open frontend/src/lib/api.ts]

"All API calls go through this helper file. fetchWithAuth attaches the cookie
credentials to every request, so the session is sent automatically.
The backend then validates it."

[Open frontend/src/context/AuthContext.tsx]

"The auth context exposes login, register, and logout functions.
After login, it refetches the session and updates the user state, which
triggers a re-render and redirects to the dashboard."

[Open frontend/src/pages/Dashboard.tsx]

"Pages are organized by feature: Dashboard, Leads, LeadForm, LeadDetail.
Each page manages its own state with useState and useEffect.
Error handling uses Mantine notifications to show toast messages when API
calls fail, so the user always knows what's happening."


PART 11: REFLECTION & CLOSING (8:30 – 9:30)
--------------------------------------------
"To wrap up, here's a quick reflection.
This project helped me understand how full-stack auth works in practice —
sessions, cookies, protected routes, and middleware.

Using Better Auth saved me from building a custom auth system, which let me
focus on the CRM logic. Prisma made database queries type-safe and
straightforward.

If I had more time, I'd add a Kanban board view for the pipeline,
a task or reminder system, and lead scoring based on activity and deal value."

"The setup is fully documented in the README with installation steps,
environment variables, database setup, and test credentials.
Thank you for watching."


RECORDING TIPS
==============
- Practice once before recording so you know where buttons are.
- Use a clean browser profile (no distracting extensions or bookmarks).
- Speak slowly — you can speed up in editing if needed.
- Show code, don't read every line. Scroll and explain the PURPOSE.
- Use Loom (free) for one-click record + shareable link.
- Keep the final video under 10 minutes.
- Test all links in incognito mode before submitting.
